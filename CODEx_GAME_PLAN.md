# Codex 开发参考文档：像素风多人在线空间游戏（参考 star-office-ui-v2）

## 1. 项目目标

我要开发一款**网页像素风多人在线空间游戏**。

风格参考：

- star-office-ui-v2
- 像素办公室 / 像素基地 / 像素公共空间
- 温暖、轻松、可互动、有在线存在感

但本项目**不是 AI agent 状态看板**，而是：

- 真人玩家进入房间
- 控制角色移动
- 与其他玩家联机互动
- 聊天、发表情、点击物件触发动作
- 完成轻量任务并获取简单奖励
- 后续可加入家具装扮、个人空间、轻经营内容

---

## 2. 项目定位

这是一个**轻社交 + 轻经营 + 多人在线像素空间**项目。

核心体验：

1. 玩家进入共享房间
2. 看到其他在线玩家
3. 控制角色移动
4. 点击家具或设施进行互动
5. 通过聊天、表情、动作建立活人感
6. 完成简单任务获得金币
7. 用金币解锁装饰、家具或功能区域

注意：

- 第一版不做战斗
- 第一版不做复杂背包
- 第一版不做大世界
- 第一版不做公会
- 第一版不做复杂经济系统
- 第一版不做 3D

第一版重点是：
**先做出“能联机、能移动、能互动、能聊天、能运行稳定”的最小可行版本。**

---

## 3. 技术栈要求

前端：

- TypeScript
- Phaser 3
- Vite

联机通信：

- Socket.IO 或 WebSocket

后端：

- Node.js
- 推荐使用 Socket.IO
- 可使用 Express 搭配

状态管理：

- 可使用 XState，或先使用清晰的本地状态结构，后续再状态机化

数据存储（第一版可选简单方案）：

- SQLite / JSON / Supabase 均可
- 第一版只要支持最基本的玩家数据和房间数据即可

---

## 4. 参考项目分析要求

Codex 需要把 star-office-ui-v2 当作**风格和结构参考**，而不是直接照抄。

需要重点借鉴：

- Phaser 3 场景组织思路
- 多场景切换思路
- WebSocket / 实时同步思路
- 角色类和远程实体渲染思路
- 状态驱动动作表现思路
- 像素风 UI 氛围

需要改造的部分：

- 把 agent 状态改成真人玩家行为
- 把状态看板逻辑改成玩家联机逻辑
- 把系统状态推送改成房间内多人同步
- 增加聊天、表情、可交互物件、任务和奖励

禁止直接把项目写成 AI agent dashboard。
必须明确是**真人玩家驱动的网页联机游戏原型**。

---

## 5. 最小可行版本（MVP）功能

第一阶段必须完成以下功能：

### 5.1 登录/进入

- 玩家输入昵称
- 进入默认公共房间
- 自动生成一个本地玩家角色

### 5.2 移动系统

- 使用键盘 WASD 或方向键移动
- 支持站立 / 行走 / 朝向变化
- 摄像机跟随本地玩家

### 5.3 多人同步

- 同一房间内可看到其他玩家
- 同步玩家进入、离开、位置、朝向、动作
- 远程玩家移动要尽量平滑，避免抖动

### 5.4 聊天系统

- 房间公共聊天
- 玩家昵称显示
- 聊天消息显示在 UI 面板
- 可先不做私聊

### 5.5 基础互动

至少实现 3 类物件：

- 桌子：触发工作动作
- 咖啡机：触发喝咖啡动作
- 沙发：触发休息动作

互动规则：

- 玩家靠近后按 E 互动，或者点击互动
- 互动后播放对应动作
- 可返回 idle 状态

### 5.6 轻量奖励

- 第一个版本先只有一种货币：coins
- 玩家完成简单互动或任务后获得少量 coins
- 暂时不做复杂数值系统

### 5.7 UI

- 玩家昵称显示在头顶
- 聊天框
- 简单的金币显示
- 简单的房间提示

## 8. 网络同步要求

Codex 必须设计清晰的多人同步协议，至少包括：

### 客户端 -> 服务端

- join_room
- move
- chat
- interact
- ping（可选）

### 服务端 -> 客户端

- room_state
- player_joined
- player_left
- player_moved
- chat_message
- interaction_state

同步要求：

- 对位置同步做节流，避免频繁发包
- 远程玩家渲染用插值或平滑过渡
- 玩家离开时要正确销毁远程实体
- 避免重复创建玩家对象
- 避免本地玩家被远程同步覆盖控制

---

## 9. 状态设计要求

角色至少包含以下状态：

- idle
- walking
- interacting
- working
- drinking
- resting

状态切换必须清晰可维护。
如果使用 XState，请保持状态机简单，不要为了“高级感”把代码写成巫术阵法。

---

## 10. 代码质量要求

Codex 输出代码时必须遵守：

- 使用 TypeScript
- 类型定义完整
- 文件拆分合理
- 不要把所有逻辑塞进一个文件
- 每个模块有明确职责
- 注释简洁但清楚
- 不要过度抽象
- 不要引入不必要的大型依赖
- 优先保证可运行

---

## 11. 调试要求（非常重要）

Codex 在每个阶段完成后，必须：

1. 安装依赖
2. 启动前端和后端
3. 打开页面进行实际调试
4. 检查浏览器控制台报错
5. 检查终端报错
6. 发现问题后自行修复
7. 修复后再次启动并验证
8. 不要只给代码不验证

必须遵循以下原则：

- 不是“我觉得能跑”
- 而是“我已经启动并检查过页面效果”
- 如果报错，先修再继续
- 如果端口冲突，自己调整
- 如果构建失败，自己修复依赖或路径问题
- 如果页面白屏，自己定位原因
- 如果联机不同步，自己排查事件收发和实体更新

---

## 12. 自动开页面调试的执行要求

Codex 必须主动做这些事，而不是等用户提醒：

- 启动前端开发服务器
- 启动后端服务
- 在浏览器中打开项目页面
- 检查首页是否正常加载
- 操作角色移动
- 检查是否有渲染异常
- 检查多人同步逻辑
- 检查聊天功能是否正常
- 检查互动功能是否正常
- 每完成一个阶段都要重新验证

如果某一步无法完成，必须明确说明是哪里卡住，并优先尝试自行修复。

---

## 13. 开发顺序要求

Codex 必须按以下顺序开发，不要跳步骤：

### 阶段 1：项目初始化

- 创建前端和后端目录
- 安装依赖
- 跑通 Vite + Phaser 3
- 跑通 Node + Socket.IO

### 阶段 2：单人场景

- 大厅场景
- 本地角色
- 键盘移动
- 摄像机跟随
- 简单动画状态

### 阶段 3：多人同步

- 进入房间
- 远程玩家显示
- 同步进入、移动、离开
- 平滑移动

### 阶段 4：聊天系统

- 聊天输入框
- 聊天消息广播
- UI 展示

### 阶段 5：互动系统

- 桌子、咖啡机、沙发
- 靠近互动
- 动作切换
- 简单奖励

### 阶段 6：轻量经济系统

- coins
- HUD 显示
- 简单奖励规则

### 阶段 7：收尾整理

- 修复 bug
- 重构明显混乱代码
- 补充 README
- 列出下一阶段建议

## 15. 严格约束

禁止：

- 擅自升级成复杂 MMO
- 一次性加入商城、公会、拍卖行、好友系统等复杂功能
- 一开始就设计复杂数据库
- 一开始就做编辑器系统
- 一开始就做微服务
- 过度抽象
- 生成大量空壳代码
- 不验证页面就宣称完成

允许：

- 先用最简单方案实现
- 后续逐步重构
- 先做能跑的 MVP，再优化结构

---

## 16. 最终目标

最终目标不是写一堆看起来很厉害的代码，而是：

**做出一个真正可以运行、可以打开页面、可以移动、可以联机、可以聊天、可以互动的网页像素风多人空间游戏原型。**

优先级永远是：

**可运行 > 可调试 > 可迭代 > 再谈优雅**
